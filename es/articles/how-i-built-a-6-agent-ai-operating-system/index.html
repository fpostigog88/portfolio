Cómo Construí un Sistema Operativo de IA de 6 Agentes

Perspectiva: Constructor / Practicante

La primera versión no se sentía como un sistema operativo. Se sentía como un conjunto de scripts inteligentes que todavía dependían de mí para recordar qué ejecutar, cuándo ejecutarlo y cómo recuperarlo si algo fallaba. Ese era el problema: si el humano sigue siendo el scheduler, el debugger y la capa de memoria, el sistema no es realmente autónomo.

La arquitectura que sobrevivió separó seis roles prácticos: intake, investigación, ejecución, verificación, memoria y notificación. Cada rol tiene un trabajo, un límite y un modo de falla. Eso terminó importando más que el modelo elegido.

La lección principal fue que la autonomía no aparece por agregar un modelo más grande. Aparece cuando el sistema tiene estado, rutas de trabajo, verificación y una forma clara de escalar tareas desde chat hacia cron o Kanban. Sin eso, la IA sigue siendo un asistente esperando instrucciones.

El valor ejecutivo está en transformar una colección de herramientas en una capacidad operativa: un sistema que captura contexto, actúa, verifica y recuerda lo estable sin convertir cada sesión en ruido.
